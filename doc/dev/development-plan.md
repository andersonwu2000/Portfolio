### 1.2 已知缺口

| 缺口 | 說明 | 影響 |
|------|------|------|
| 整合測試為空 | `tests/integration/` 僅有 `__init__.py` | API 端到端行為無自動化驗證 |
| 策略數量少 | 僅 Momentum 和 Mean Reversion 兩套策略 | 策略多樣性不足 |
| 無即時行情 | 僅依賴 Yahoo Finance 歷史資料 | 無法支援 paper/live 模式的即時交易 |
| 無券商串接 | `live` 模式尚未對接真實券商 API | 無法進行實盤交易 |
| Web 前端無測試 | 無 Vitest 或 E2E 測試 | 前端行為無自動化保障 |
| 效能未優化 | 大型回測（多標的、長週期）可能耗時過長 | 使用者體驗受影響 |

## Phase 1：測試補強與品質基礎（第 1-2 月）

> **目標：** 建立自動化測試覆蓋，確保後續開發有信心重構與擴充。

### 1.1 後端整合測試

**優先級：高** | **預估工時：2 週**

| 任務 | 說明 |
|------|------|
| API 端到端測試 | 使用 `httpx.AsyncClient` + `TestClient` 測試所有 REST 端點 |
| 回測完整流程測試 | 從 API 提交到結果查詢的完整流程 |
| 認證測試 | JWT 登入/登出、角色權限、無效 Token 拒絕 |
| WebSocket 測試 | 連線、訂閱、推送、斷線重連 |
| 風控整合測試 | 訂單提交 → 風控攔截 → 回應驗證 |

**驗收標準：**
- `tests/integration/` 下至少 20 個測試
- 覆蓋所有 API 端點的正常與異常路徑
- CI 中整合測試全部通過

### 1.2 Web 前端測試

**優先級：中** | **預估工時：1.5 週**

| 任務 | 說明 |
|------|------|
| Vitest 設定 | 設定 `vitest.config.ts`，整合 `@testing-library/react` |
| Hook 測試 | `useApi`、`useBacktest`、`useDashboard` 核心 Hook |
| 工具函式測試 | 格式化函式（貨幣、百分比、日期） |
| 頁面快照測試 | 各頁面基本渲染測試 |

**驗收標準：**
- `make web-test` 指令可執行
- 核心 Hook 與工具函式覆蓋率 > 80%

### 1.3 效能基準測試

**優先級：中** | **預估工時：1 週**

| 任務 | 說明 |
|------|------|
| 回測效能基準 | 記錄不同標的數量（5/20/50）× 不同時間範圍（1年/3年/5年）的執行時間 |
| 記憶體追蹤 | 大型回測的記憶體使用量剖析 |
| 瓶頸識別 | `cProfile` 分析熱點函式 |

**產出：** 效能基準報告，作為後續優化的參照點

## Phase 2：策略框架與研究工具（第 2-3 月）

> **目標：** 擴充策略庫，建立因子研究框架，提升回測效能。

### 2.1 新增策略模板

**優先級：高** | **預估工時：3 週**

| 策略 | 類型 | 說明 |
|------|------|------|
| RSI Oversold | 技術面 | RSI 低於閾值時買入，超買時賣出 |
| MA Crossover | 技術面 | 雙均線交叉策略 |
| Pairs Trading | 統計套利 | 配對交易，利用價差回歸 |
| Factor Investing | 多因子 | 結合多個因子的綜合策略 |
| Sector Rotation | 板塊輪動 | 基於動量的板塊輪動配置 |

每套策略包含：
- 策略實作（`strategies/` 目錄）
- 單元測試
- 預設參數與說明文件

### 2.2 因子研究框架

**優先級：高** | **預估工時：2 週**

| 任務 | 說明 |
|------|------|
| 因子分析 CLI | `python -m src.cli.main factor-analysis --factor momentum --universe ...` |
| IC 計算 | 因子值與未來報酬的 Information Coefficient |
| 因子衰減分析 | 不同持倉週期的因子有效性 |
| 因子合成 | 多因子加權組合（等權、IC 加權） |

### 2.3 回測效能優化

**優先級：中** | **預估工時：2 週**

| 任務 | 說明 |
|------|------|
| 資料載入優化 | 向量化價格查詢，避免逐 symbol 迴圈 |
| Polars 評估 | 評估 Polars 取代 Pandas 的可行性與效能提升幅度 |
| 快取強化 | 回測中間結果快取（因子值、價格矩陣） |
| 並行回測 | 多策略/多參數回測的並行執行 |

**驗收標準：**
- 20 檔標的 × 3 年回測時間 < 30 秒
- 50 檔標的 × 5 年回測不因記憶體不足而中斷

### 2.4 回測報告強化

**優先級：中** | **預估工時：1 週**

| 任務 | 說明 |
|------|------|
| HTML 報告 | 生成包含 NAV 曲線圖、回撤圖、持倉變化的 HTML 報告 |
| 基準比較 | 支援與大盤指數（SPY、0050.TW）的績效比較 |
| 交易明細 | 輸出完整交易記錄 CSV |

## Phase 3：即時交易基礎建設（第 3-4 月）

> **目標：** 打通即時行情與券商 API，實現 paper trading。

### 3.1 即時行情對接

**優先級：高** | **預估工時：3 週**

| 任務 | 說明 |
|------|------|
| 行情 DataFeed 介面 | 擴充 `DataFeed` ABC 支援即時行情訂閱 |
| 台灣券商 API | 串接富邦/元大 API（或 TWSE 公開資料） |
| 行情推送 | 透過 WebSocket `market` 頻道推送至前端 |
| 快取與斷線處理 | 行情快取 + 自動重連機制 |

### 3.2 Paper Trading 模式

**優先級：高** | **預估工時：3 週**

| 任務 | 說明 |
|------|------|
| 策略排程器 | 定時觸發策略 `on_bar()`，依再平衡頻率執行 |
| PaperBroker 完善 | 使用即時價格模擬成交，記錄至資料庫 |
| 持倉持久化 | Portfolio 狀態寫入資料庫，重啟後恢復 |
| 績效追蹤 | 即時計算 P&L、NAV、各項績效指標 |

**驗收標準：**
- `QUANT_MODE=paper` 啟動後可自動執行策略
- 前端即時顯示持倉與損益變化
- 重啟後持倉狀態正確恢復

### 3.3 告警通知

**優先級：中** | **預估工時：1.5 週**

| 任務 | 說明 |
|------|------|

## Phase 4：進階功能與生產準備（第 4-6 月）

> **目標：** 實盤交易對接、監控強化、系統穩定性提升。

### 4.1 實盤交易對接

**優先級：高** | **預估工時：4 週**

| 任務 | 說明 |
|------|------|
| BrokerAdapter 實作 | 對接台灣券商下單 API（富邦/元大） |

- `QUANT_MODE=live` 可透過券商 API 下單

## 四、技術決策紀錄

### 4.1 已定案

| 決策 | 選擇 | 理由 |
|------|------|------|
| 金額型別 | `Decimal` | 避免浮點精度問題，金融計算必須精確 |
| 策略回傳格式 | 目標權重 dict | 簡化策略開發，由引擎負責訂單轉換 |
| 風控規則設計 | 純函式工廠 | 無狀態、可測試、可組合 |
| 前端共享 | @quant/shared + 平台 adapter | 避免重複型別定義，保持平台特定邏輯隔離 |
| 資料庫 ORM | SQLAlchemy Core（非 ORM） | 與 Alembic 共用 metadata，保持查詢透明度 |
| 認證方式 | JWT + API Key | JWT 用於前端 Session，API Key 用於程式化存取 |

### 4.2 待決策

| 決策 | 選項 | 評估時機 |
|------|------|----------|
| DataFrame 引擎 | Pandas vs Polars | Phase 2 效能優化時評估 |
| 券商 API | 富邦 vs 元大 vs 自建 | Phase 3 啟動前確認 |
| 即時行情來源 | 券商 API vs TWSE 公開資料 vs 第三方 | Phase 3 啟動前確認 |
| 訊息佇列 | 無（目前）vs Redis vs RabbitMQ | 多用戶場景時評估 |

## 五、風險管理

| 風險 | 機率 | 影響 | 緩解措施 |
|------|------|------|----------|
| 券商 API 變更或不穩定 | 中 | 高 | 抽象 BrokerAdapter 介面，可快速切換券商 |
| Yahoo Finance 限流或停服 | 中 | 中 | 已有 24h 本地快取；規劃備用資料源 |
| 回測效能瓶頸 | 中 | 中 | Phase 1 建立基準，Phase 2 針對性優化 |
| 即時交易資金安全 | 低 | 高 | 多層風控 + Kill Switch + 每日金額上限 + 實盤啟動確認 |
| 前後端型別不一致 | 低 | 中 | @quant/shared 統一型別定義 + CI 型別檢查 |

## 六、開發流程規範

### 6.1 分支策略

```
main ──────────────────────────────────────────────►
  │                                    │
  ├── feature/phase1-integration-tests ├── merge
  ├── feature/strategy-rsi             ├── merge
  └── feature/paper-trading            └── merge
```

- 每個任務建立 feature branch
- PR 需通過 CI（lint + test + typecheck）後方可合併
- 合併前需至少一次 self-review

### 6.2 品質門檻

| 檢查項 | 標準 |
|--------|------|
| 後端測試 | `make test` 全部通過 |
| 程式碼品質 | `make lint`（ruff + mypy strict）無警告 |
| Web 型別 | `make web-typecheck` 通過 |
| Mobile 型別 | `make mobile-typecheck` 通過 |
| 新功能 | 附帶對應測試 |

### 6.3 文件維護

- 新增 API 端點同步更新 `doc/api-reference-zh.md`
- 新增策略同步更新 `doc/user-guide-zh.md` 的策略章節
- 架構性變更同步更新 `CLAUDE.md`
